# Contest IO

[![Go Version](https://img.shields.io/github/go-mod/go-version/aaa2ppp/contestio)](go.mod)
[![License](https://img.shields.io/github/license/aaa2ppp/contestio)](LICENSE)

**Contest IO** - удобный ввод-вывод, который не тормозит. Специально для контестов на Go.

> **Статус:** Бета-версия. Твои замечания и PR делают библиотеку лучше.

## Почему Contest IO?

**Ты пришел побеждать!** А не писать ввод/вывод.

`fmt`? Комфортный седан. Довезет, но с аллокациями.  
`bufio.Scanner`? Шустрее, без аллокаций. Но на смешанных данных - танцы с бубном.  
`bufio.Reader`? Быстро. Но циклы и парсинг - сам знаешь.

**Contest IO - это спорткар.**  
Максимум скорости. Минимум кода.

- Ноль аллокаций - только твои данные.
- Токенизатор обгоняет `bufio.Scanner` в 1.5 раза.
- Парсер целых не уступает `strconv.Atoi`. Любые целые, любой разрядности.
- Простой интерфейс: `ScanInt/PrintInt` для целых, `ScanFloat/PrintFloat` для float.
- Ввод/вывод массивов чисел одной строчкой.
- **Киллер-фича:** `contestio-inline` встроит код прямо в твое решение - никаких зависимостей.

**Готов испытать Contest IO в деле?**

## Быстрый старт

### Установка

```sh
go get github.com/aaa2ppp/contestio@latest
```

### Читаем массив, обрабатываем, выводим

```go
package main

import (
    "os"
    . "github.com/aaa2ppp/contestio" // Точечный импорт делает код кратким
)

func main() {
    br := NewReader(os.Stdin)
    bw := NewWriter(os.Stdout)
    defer bw.Flush()

    // Читаем размер массива
    var n int
    if _, err := ScanIntLn(br, &n); err != nil {
        panic(err) // В задачах ввод всегда корректен, поэтому можно падать
    }

    // Читаем массив (одна строчка!)
    a := make([]int, n)
    if _, err := ScanInts(br, a); err != nil {
        panic(err)
    }

    // Твое решение
    for i := range a {
        a[i] *= a[i]
    }

    // Выводим результат (тоже одна строчка)
    PrintIntsLn(bw, a)
}
```

### Чтение строк `ScanBytes/ScanString`

```go
var err error
for {
    s, err := ScanString('\n') // прочитает строку и обрежет финальные пробелы
    if err != nil {
        break
    }
    // обрабатываем строку...
}
if err != io.EOF {
    panic(err)
}
```
> Если нужны байты, используй `ScanBytes`. Обе функции читают до разделителя или конца файла,
> Обрезают разделитель и финальные пробелы. Возвращают копию данных (можешь безопасно пользоваться). 
> Размер строки ограничен только доступной оперативкой.

> Чтобы не обрабатывать ошибки вручную, используй тег `must`. Функции библиотеки будут паниковать 
> на любой ошибке, кроме `io.EOF`.
> ```bash
> go run -tags=must main.go
> ```

### Единый интерфейс для разных числовых типов
```go
// Читаем количество элементов
var n int
ScanInt(br, &n)

// Читаем слайсы разных интов одной и той же функцией
a := make([]uint8, n)
ScanInts(br, a)

b := make([]int32, n)
ScanInts(br, b)

c := make([]int64, n)
ScanInts(br, c)

// Для чисел с плавающей - ScanFloats
d := make([]float32, n)
ScanFloats(br, d)

e := make([]float64, n)
ScanFloats(br, e)

// Выводим
PrintIntsLn(bw, a)
PrintIntsLn(bw, b)
PrintIntsLn(bw, c)
PrintFloatsLn(bw, d)
PrintFloatsLn(bw, e)
```

### Чтение всех чисел из строки (неизвестная длина)

```go
// Преаллоцируем память под данные
a := make([]int32, 1<<20) // 1M
a, _ = ScanIntsLn(br, a[:0]) // функция поймет с каким типом работает

// Аллоцирует сама, но нужно явно указать тип
b, _ := ScanIntsLn[[]int32](br, nil)
```

### Вывод с разделителями

```go
op := WO{Begin: "[", Sep: ", ", End: "]\n"}
PrintInts(bw, op, []int32{1, 2, 3, 4}) // [1, 2, 3, 4]
```

### Чтение/вывод матрицы
```go
var rows, cols int
ScanIntLn(br, &rows, &cols) // Читаем обе переменные за раз

matrix := make([][]int, rows)
for i := range matrix {
    matrix[i] = make([]int, cols)
    ScanInts(br, matrix[i]) // Читаем строку матрицы
}

// Выводим
for _, row := range matrix {
    PrintIntsLn(bw, row)
}
```

### Смешанный ввод/вывод (contestio/bufio/fmt)
```go
var age int
var name string

ScanIntLn(br, &age)           // Читаем число
name, _ = br.ReadString('\n') // Строку стандартным методом bufio.Reader
name = strings.TrimSpace(name)

// Выводим с помощью fmt
fmt.Fprintf(bw, "name: %q, age: %d\n", name, age)
```

## Утилита `contestio-inline`:

Многие проверяющие платформы (Яндекс.Контест, Codeforces) запрещают внешние зависимости.  
Можно держать шаблон со всеми парсерами и копипастить в каждое решение - громоздко и неудобно.

**`contestio-inline`** решает эту проблему:

- Ты пишешь и тестируешь решение, используя точечный импорт `contestio` - код чистый и краткий.
- Перед отправкой выполняешь одну команду - утилита добавит в твой файл только те функции библиотеки, которые реально используются.
- Полученный файл самодостаточен и готов к отправке на проверку.
- Утилита не трогает твой исходный файл - она создает новый, а старый переименует, добавив в конец '~'.

```sh
# Установка
go install github.com/aaa2ppp/contestio/cmd/contestio-inline@latest

# Инлайн (работает только с точечным импортом!)
contestio-inline main.go

# Файл main.go теперь содержит весь необходимый код, main.go~ - оригинальный код.
# Для отката: contestio-inline -clear main.go
```

**Важно:**
- Утилита требует точечный импорт (`import . "github.com/aaa2ppp/contestio"`). Обычный или алиасный импорт не поддерживается.
- Утилита поддерживает флаг тегов сборки `-tags`. Не забывай указывать, если нужно.
- `contestio-inline -clear` именно вычищает код библиотеки, а не переименовывает старый `main.go~`, на случай, если после инлайна ты делал правки.
  Старый `main.go~` будет перезаписан! Позаботься о бэкапе заранее.
- **Если после инлайна или очистки код не компилируется** (забыл теги, ошибка в коде или я накосячил) - утилита ничего не меняет. 
  Твой исходник в безопасности. На крайний случай есть бэкап `main.go~`.

## Производительность

Все бенчмарки - для **1M (2^20) чисел** (`int` или `float64`), буфер 4 КБ.  
**Система:** Intel Core i3-7100 @ 3.90GHz, Windows, Go 1.24.2.

В таблицах **жирным** выделены core функции `contestio`.  
Время в миллисекундах - чем меньше, тем лучше.

### Что значат цифры?

- **`fmt.Fscan`** - универсален, для каждого значения создает интерфейс. 1 млн чисел - 1 млн аллокаций.
- **`ReadString + Fields`** - сначала должен все прочесть.
- **`Scanner + ScanWords`** - универсальность съедает время.
- **`AppendXXX`** - пишет напрямую в буфер, но иногда места уже нет (но это решаемо).
- **Функции `contestio`** - парсят и пишут числа напрямую в буфер. 

> **Почему Contest IO не обгоняет `AppendInt`?**  
> Потому что под капотом использует его. В бенчмарках `printSlice` на 2 ms медленнее `AppendInt`.
> Это плата за возможность работать с любыми числовыми слайсами и разделители.  

### Выводы

- `fmt` - когда не торопишься и не жалко ресурсов
- `Scanner` + `Append*` - хорошая пара, но требует ручного кода.
- **Contest IO** - дает скорость и ноль аллокаций с минимумом усилий.

## Расширенное использование: API с интерфейсами (тег `sugar`)

Если хочешь абстрагироваться от конкретных типов и использовать интерфейсы - включи сахар. 
Он активируется тегом сборки `-tags=sugar` и добавляет обертки над слайсами `Ints[T Int]` и `Floats[T Float]`,
которые реализуют интерфейсы для универсальных функций `ScanSlice` и `PrintSlice`.

**Пример с "сахаром":**

```go
package main

import (
    "os"
    . "github.com/aaa2ppp/contestio"
)

func solve(a []int) []int {
    for i := range a {
        a[i] *= a[i]
    }
    return a
}

func main() {
    br := NewReader(os.Stdin)
    bw := NewWriter(os.Stdout)
    defer bw.Flush()

    var n int
    var a Ints[int] // Используем сахарный тип

    if _, err := ScanIntLn(br, &n); err != nil {
        panic(err)
    }
    
    a = Resize(a, n)
    if _, err := ScanSlice(br, a); err != nil { // Читаем через общий интерфейс
        panic(err)
    }

    a = solve(a)

    PrintSliceLn(bw, a) // Вывод через общий интерфейс
}
```

**Особенности:**
- Позволяет писать более абстрактный код, не привязываясь к конкретным типам срезов, отказавшись от специализированных функций вроде `ScanInts`, `PrintInts` и т.п.
- Использует интерфейсные обертки над срезами. Важно: их можно прозрачно передавать, принимать и работать, как с обычным срезом.
- Требует указания тега сборки при компиляции: `go build -tags=sugar`.
- Для комфортной работы в VS Code необходимо добавить `"go.buildTags": "sugar"` в настройки проекта (иначе автодополнение и тесты не увидят код с тегом).

Сахар **не обязателен** для эффективного использования библиотеки. Основной API (`ScanInts`, `PrintInts` и т.п.) справляется со всем, что нужно на контестах. 
Сахар - это дополнительный слой для тех, кто хочет поэкспериментировать с интерфейсами или хочет использовать библиотеку вне соревнований.

## Обработка ошибок: тег сборки `must`

По умолчанию все функции сканирования и печати возвращают ошибки.  
В задачах с гарантированно корректным вводом это приводит к шаблонным проверкам `if err != nil { panic(err) }`.  

Чтобы избавиться от них, добавь тег сборки `must`:

```sh
go build -tags=must
```

При активном теге `must` любая ошибка (кроме `io.EOF`) вызывает панику.  
Это касается всех публичных функций `ScanXXX/PrintXXX` и их вариантов с `Ln`.

Без тега поведение остается прежним - ошибки возвращаются.

**Пример:**
```go
// Без тега must
n, err := ScanInt(br, &x)
if err != nil {
    panic(err)
}

// С тегом must (код короче)
ScanInt(br, &x) // паникует при ошибке
```

## Настройка VS Code для работы с тегами сборки

Если ты используешь «сахарный» API или тег `must`, добавь в `.vscode/settings.json`:

```json
"go.buildTags": "sugar,must"
```

После этого автодополнение и тесты будут видеть код с тегом.
